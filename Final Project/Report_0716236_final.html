Wi-Fi 環境下影音串流品質預測

透過 Wi-Fi Tx rate 來預測當下最佳的影音串流品質

Name: 劉耀文 (Yao-Wen Liu)

Student ID: 0716236

I. Introduction

這個Project之題目發想來自於目前正在進行的專題研究：利用Wi-Fi邊緣運算架構進行主動式且通道感知的影像串流服務品質調適，目前專題是使用

  • NSS
  • MCS index
  • Guard Interval
  • 實驗裝置占用airtime比例之moving average
  • 同Wi-Fi環境下其他裝置所占用airtime比例之moving average

以及固定前三項參數下全速傳輸所能達到的速率最大實驗值，來計算及推測目前實驗裝置最適合的影音串流品質為何，並主動調整該串流品質，降低傳統客戶端須向伺服器回報速率下降、沒收到封包導致之服務中斷問題，大大提升使用者體驗，未來可能會加上QoE metrics以及搭配類神經網路來做為最終之模型，將使用者體驗最佳化。

經過了將近一學期的機器學習概論課程，儘管目前result的計算過程不會太過複雜，但想試試看能不能用簡單的分類模型來準確預測當下最佳之串流品質。

II. Data Collection

蒐集資料

使用搭載MediaTek MT7622處理器以及MediaTek MT7915網路介面卡之Wi-Fi路由器作為實驗AP，並於此安裝含有驅動程式之OpenWrt系統。

  1. 修改AP之驅動程式：使得Tx rate以及airtime可以透過debugfs之檔案輸出，重新build新的OpenWrt image後透過serial port以及U-boot介面刷入AP當中。
  2. 建立簡單的publish/subscriber程式：當作AP以及edge system的溝通橋樑
    • publisher端：將AP之debugfs系統中的輸出檔案做前處理後發布
    • subscriber端：接收來自發布端的資料，並做一些計算後得到目前最佳的串流品質
    • 使用python3-paho-mqtt套件
    • Source code
  3. 修改subscriber端之程式碼：使得該程式可以將我們所需的參數以及計算得到之串流品質輸出至檔案，以利後續project使用。
  4. 產生資料：在實驗裝置上運行串流服務，並透過iperf3製造其他裝置和AP間的流量以及裝置和AP間的不同距離來產生不同狀況下的實驗數據。

資料內容

  • NSS - 傳輸使用天線個數 (binary: NSS1 or NSS2)
  • MCS index - MCS參數 (nominal: MCS0-MCS9)
  • Guard Interval - 資料幀傳輸間隔 (binary: SGI or GI)
  • device airtime - 實驗裝置所佔用airtime比例之moving average (numeric: 0.0-1.0)
  • other airtime - 其他裝置所佔用airtime比例之moving average (numeric: 0.0-1.0)
  • quality - 當下之影音串流品質 (binary: 1080p or 720p)

III. Preprocessing

Data Visualization

雖然說所有feature都是做visualization，但dataset裡面有不同種類的feature，分別用是bar chart畫出categorical feature、用histogram畫出numerical feature。

  • 使用matplotlib套件之bar()畫出categorical feature之bar chart
  • 使用matplotlib套件之hist()畫出numerical feature之histogram

由上圖可以觀察到：

  • quality似乎和NSS有很大的關係：NSS1的樣本數跟720p的樣本數相差不大
  • quality似乎也和other_airtime有很大的關係：other_device airtime以0.2作為劃分，樣本數和quality的分布相差不大，推測是因為其他裝置占了太多airtime，壓縮了實驗裝置的速率。
  • GI的分布跟quality似乎沒太大關係：可能是因為GI只是data frame傳輸間隔，對整體傳輸速率不會有顯著影響。
  • MCS index集中在MCS4-MCS9：推測是因為產生數據時，實驗裝置和AP距離不會相差太遠，所以比較難產生MCS0-MCS3的資料。
  • device_airtime會集中在0-0.2：推測是因為串流服務使用之網路流量不會太多。
  • other_airtime最大值不超過0.8：是因為Wi-Fi在傳輸時不會所有時間都在傳輸封包，會有一些overhead，如：packet header，而導致這種狀況。

Data Preprocessing

因為資料在輸出前已經經過了不同的資料前處理，故僅需在讀入資料後將NSS、MCS index、GI以及video quality做簡單的label encoding，將字串轉換為integer即可。

  • 使用pandas套件之 read_csv() 讀入資料
  • 使用scikit-learn套件之 LabelEncoder() 達到label encoding的效果

IV. Models

因為dataset的feature有numerical以及categorical兩種，故選擇了decision tree、random forest以及K-NN作為此project的model，並使用K-Fold cross-validation作為判斷模型訓練成果之依據。

  • 使用sklearn套件之KFold()當作K-Fold cross-validation
    • K=3



# get index of each epoch
k_fold = KFold(n_splits=3)
k_fold.get_n_splits(X)
  • 使用sklearn套件之DecisionTreeClassifier()當作是decision tree之訓練模型
    • 使用entropy當作node splitting之依據



clf = DecisionTreeClassifier(criterion="entropy")
clf.fit(x_train, y_train)
y_predict = clf.predict(x_test)
  • 使用sklearn套件之RandomForestClassifier()當作是random forest之訓練模型
    • 模型內有10棵決策樹參與投票
    • 每棵樹最多使用sqrt(5)個feature訓練
    • 使用entropy當作node splitting之依據
    • 模型預設使用boostrap



clf = RandomForestClassifier(n_estimators=10, max_features="sqrt", criterion="entropy")
clf.fit(x_train, y_train)
y_predict = clf.predict(x_test)
  • 使用sklearn套件之KNeighborsClassifier()當作是K-NN之訓練模型
    • 找出最近10個sample，並以距離之倒數當作投票權重



clf = KNeighborsClassifier(n_neighbors=10, weights="distance")
clf.fit(x_train, y_train)
y_predict = clf.predict(x_test)

V. Results

Confusion Matrix

  • 使用sklearn套件之confusion_matrix()得到confusion matrix
  • 使用sklearn套件之ConfusionMatrixDisplay()畫出confusion matrix

Scores

  • 使用sklearn套件之accuracy_score()、recall_score()以及precision_score()當作計算performance的工具
Accuracy Sensitivity Precision
Decision Tree 93.99% 92.94% 93.28%
Random Forest 95.63% 90.18% 100.0%
K-NN with K=10 90.71% 88.85% 89.89%

觀察以上數據，可以發現此dataset使用random forest模型可以獲得最好的表現，緊接著是decision tree，最差的是K-NN，可以推測：

  • Random forest以及decision tree會最佳，可能是因為某些feature跟label的分佈有很大的相關性，且在訓練過程中會選擇最佳的feature當作是第一個分裂點使用的feature，所以這兩個模型可以有比較好的表現。
  • K-NN表現較差則是因為所有feature都考慮到了，所以像是GI這種比較沒有影響的參數也被考慮進去，所以讓表現可能會稍微差一些。

VI. Conclusion

若要用這些模型來預測接下來的結果，只需要從AP的debugfs下之檔案找出NSS、MCS index、airtime等參數輸入至模型即可，或許能將機器學習模型整合到publisher/subscriber程式當中。

儘管目前專題不需要用到機器學習模型來解決問題，但經過了這次project可以發現使用機器學習的效果不錯，所以證明這個dataset確實是可以使用機器學習來預測串流服務的品質。

再加上之後會在預測服務品質的module加上QoE metrics，以最大化使用者體驗，如果是用單純的算式計算，可能會無法得出最佳結果，這時候機器學習模型就可以派上用場了，可以先把各種參數記錄下來及計算QoE metrics後，再人工標記資料的label並當作模型訓練的資料，最後透過模型預測最佳串流品質。

結論就是儘管現在仍看不出這些模型的優勢，但後續的研究讓預測module變得更加複雜後，若引入機器學習的作法，肯定會讓整體效率增加不少，所以這次的project可以說是專題研究的先行研究，增加接下來的研究效率。
